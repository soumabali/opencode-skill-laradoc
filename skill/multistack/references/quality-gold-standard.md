# Quality Gold Standard (Persistent Quality Blueprint)

Purpose: make output quality consistent with a high documentation benchmark.
This standard is mandatory for MAX profile unless user explicitly asks otherwise.

## 1) Tooling and Build Stack (non-negotiable)

- Use **native Docusaurus workflow** end-to-end.
- Use Docusaurus i18n (`id` + `en`) with real locale routes.
- Do **not** replace Docusaurus with ad-hoc static HTML converter in MAX mode.

Minimum artifacts:
- `docusaurus.config.js|ts`
- `sidebars.js|ts`
- `docs/` content tree
- `i18n/en/...` content tree
- `build/` generated by Docusaurus build

## 2) Documentation Breadth (beyond core 6 pages)

Core pages are required, plus expanded pages for operational depth:
- onboarding-checklist
- onboarding-junior / day-1-day-3
- module-map / controller-map
- api-endpoints / api-full-catalog / request-response-catalog
- database-schema-map / database-full-catalog / domain-model
- business-flows / end-to-end-process / state-machine
- role-access-matrix / verification-matrix / verification-guideline
- deployment-runbook / observability-ops / common-maintenance-tasks
- changelog / glossary

If a page is not applicable, keep placeholder with `verification required` and reason.

## 3) Writing Style and Detail

- Evidence-first, file-path anchored.
- Practical onboarding style: explain where to start, what to check, and common pitfalls.
- Keep bilingual parity: EN page and ID page should carry the same intent and structure.
- Keep technical language precise and concise (no marketing tone).
- Preserve page-specific writing patterns:
  - onboarding pages: step-by-step and action-oriented
  - API pages: endpoint/cookbook/catalog oriented
  - DB pages: schema/domain mapping oriented
  - ops pages: runbook/checklist oriented
  - verification pages: audit/matrix oriented

## 4) Multi-pass Production Flow

Use phased execution to avoid one-pass shallow output:

1. Full-scan pass (inventory and evidence map)
2. Generation pass (core + expanded docs)
3. Polish pass (consistency, readability, navigation)
4. Crosscheck pass (coverage, links, quality gates)
5. Deploy pass (pages + domain verification)

Each phase must produce a short status artifact/checklist.

## 5) Verification and Deploy Discipline

Before deploy, all must pass:
- Docusaurus build green
- i18n routes green (`/docs/...` and `/en/docs/...`)
- quality gate pass
- quality floor pass
- consistency contract pass

After deploy, verify:
- custom domain root 200
- one ID docs page 200
- one EN docs page 200
- report URLs in strict format

## 6) Strict Reporting Format

Use this exact output block for final report:
- STATUS
- DONE
- FILES CHANGED
- COMMIT/PUSH STATUS
- QUALITY SUMMARY
- BLOCKER
- ETA

If any critical deliverable is missing (mandatory docs, mandatory sections, parity checks, quality score threshold), set STATUS=FAIL and block deploy.
